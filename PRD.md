## 2. 产品背景

当前需要快速开发一个类似 ChatGPT 的网页对话问答 Demo，用于验证核心交互体验和大模型调用链路。该 Demo 不追求完整商业产品能力，重点是尽快跑通从用户输入、后端调用大模型、前端流式展示回答到本地保存会话的最小闭环。

## 3. 产品目标

### 3.1 核心目标

- 提供一个可以直接使用的网页聊天界面。
- 支持用户输入问题并获得大模型回答。
- 支持助手回答流式输出，降低等待感。
- 支持简单多轮上下文对话。
- 支持本地会话保存，刷新页面后可恢复历史。
- 保证模型 API Key 不暴露在前端。

### 3.2 非目标

当前 Demo 阶段不实现以下能力：

- 用户登录 / 注册。
- 第三方登录。
- 文件上传。
- 文档解析。
- 知识库问答 / RAG。
- 管理后台。
- 权限管理。
- 计费系统。
- 团队协作。
- 数据库持久化。
- 复杂 Agent 或工具调用。

## 4. 用户画像

### 4.1 Demo 体验用户

- 需要快速体验 AI 对话功能的业务方、客户或内部团队成员。
- 关注页面是否像一个可用的对话产品。
- 关注回答速度、流式体验和多轮对话效果。

### 4.2 开发和评估人员

- 需要基于 Demo 判断后续技术路线是否可行。
- 关注接口是否稳定、模型调用是否顺畅、架构是否方便扩展。

## 5. 使用场景

### 5.1 普通问答

用户打开页面，在输入框中输入一个问题，点击发送或按 Enter，系统开始流式返回回答。

### 5.2 多轮追问

用户在同一个会话内继续追问，系统结合当前会话上下文生成回答。

### 5.3 查看历史会话

用户刷新页面后，可以继续看到之前保存在浏览器 localStorage 中的会话记录。

### 5.4 新建会话

用户点击新建会话，系统创建一个空白会话，后续问题不再携带旧会话上下文。

### 5.5 停止生成

当助手正在生成回答时，用户可以点击停止按钮，中断当前流式输出。

## 6. 功能需求

## 6.1 聊天主界面

### 6.1.1 功能描述

页面打开后展示一个单页聊天工作台，布局类似 ChatGPT：

- 左侧为会话列表。
- 中间为消息列表。
- 底部为输入区。

### 6.1.2 需求明细

- 页面无需登录即可进入。
- 默认创建一个空会话。
- 当前无消息时展示空状态。
- 用户发送第一条消息后，空状态消失。
- 页面刷新后恢复 localStorage 中的会话数据。

### 6.1.3 验收标准

- 打开页面后无需任何账号操作即可输入问题。
- 页面包含会话列表、消息展示区和输入区。
- 刷新页面后，本地已有会话可以恢复。

## 6.2 消息输入

### 6.2.1 功能描述

用户可以通过底部输入框输入问题并发送给模型。

### 6.2.2 需求明细

- 输入框支持多行文本。
- Enter 发送消息。
- Shift + Enter 换行。
- 输入为空或全是空白字符时不可发送。
- 正在生成回答时，发送按钮进入不可重复提交状态。
- 正在生成回答时，显示停止生成按钮。

### 6.2.3 验收标准

- 输入普通文本后按 Enter 可以发送。
- 按 Shift + Enter 不发送，而是在输入框中换行。
- 空内容不能发送。
- 生成过程中不能重复发送同一条消息。
- 点击停止生成后，当前回答停止追加内容。

## 6.3 消息展示

### 6.3.1 功能描述

聊天区展示用户消息和助手消息，并支持 Markdown 内容。

### 6.3.2 需求明细

- 用户消息和助手消息使用不同视觉样式。
- 消息按时间顺序展示。
- 新消息出现时，消息列表自动滚动到底部。
- 助手回答支持 Markdown 渲染。
- 助手回答支持代码块高亮。
- 助手消息支持复制内容。
- 模型生成中时，助手消息实时追加文本。

### 6.3.3 验收标准

- 用户消息和助手消息可以清晰区分。
- 模型回答中的标题、列表、代码块可以正常显示。
- 代码块样式可读。
- 点击复制后可以复制助手回答文本。
- 流式输出过程中内容逐步出现，而不是一次性出现。

## 6.4 流式回答

### 6.4.1 功能描述

前端调用后端流式接口，后端调用大模型接口，并将模型增量输出转发给前端。

### 6.4.2 需求明细

- 前端通过 `POST /api/chat/stream` 发送当前会话消息。
- 后端从环境变量读取模型 API Key。
- 后端调用大模型流式接口。
- 后端向前端返回 SSE 或 chunked stream。
- 前端逐块读取内容并追加到当前助手消息。
- 流式结束后标记消息完成。
- 如果请求失败，前端展示错误提示。

### 6.4.3 验收标准

- 用户发送问题后，页面可以看到回答逐步输出。
- 后端 API Key 不会出现在前端代码或浏览器请求参数中。
- 模型调用失败时，页面有明确错误提示。
- 流式结束后，当前助手消息状态变为完成。

## 6.5 多轮上下文

### 6.5.1 功能描述

系统支持在同一会话内进行简单多轮对话。

### 6.5.2 需求明细

- 前端发送请求时携带当前会话最近若干条消息。
- 后端对消息数量或文本长度做基础裁剪，避免上下文过长。
- 新建会话后不携带旧会话上下文。

### 6.5.3 验收标准

- 用户可以基于上一轮回答继续追问。
- 至少支持 5 轮连续上下文对话。
- 新建会话后，模型不应依赖旧会话内容回答。

## 6.6 会话管理

### 6.6.1 功能描述

Demo 阶段提供轻量会话管理，用于演示多会话能力。

### 6.6.2 需求明细

- 支持新建会话。
- 支持切换会话。
- 会话标题默认取第一条用户消息前若干字符。
- 当前会话消息变化后自动保存到 localStorage。
- 支持删除会话。

### 6.6.3 验收标准

- 点击新建会话后进入空白聊天。
- 点击历史会话后能看到对应消息。
- 第一条用户消息发送后，会话标题自动更新。
- 删除会话后，该会话从列表消失。
- 刷新页面后，会话列表仍可恢复。

## 6.7 错误处理

### 6.7.1 功能描述

当模型调用、网络请求或服务异常时，系统应给用户明确反馈。

### 6.7.2 需求明细

- 后端模型调用失败时返回错误事件。
- 前端展示错误提示，不让页面卡死。
- 请求失败后输入框恢复可用。
- 停止生成后输入框恢复可用。

### 6.7.3 验收标准

- 关闭后端或配置错误 API Key 时，前端能显示失败提示。
- 失败后用户可以继续重新发送。
- 停止生成后用户可以继续输入新问题。

## 7. 接口需求

## 7.1 健康检查接口

### 请求

`GET /api/health`

### 响应

```json
{
  "status": "ok"
}
```

### 验收标准

- 后端启动后访问该接口返回 `status: ok`。

## 7.2 流式聊天接口

### 请求

`POST /api/chat/stream`

### 请求体

```json
{
  "messages": [
    {
      "role": "user",
      "content": "你好"
    }
  ],
  "model": "gpt-4.1-mini",
  "temperature": 0.7
}
```

### 字段说明

- `messages`：当前会话上下文消息，必填。
- `messages[].role`：消息角色，可选值为 `user`、`assistant`、`system`。
- `messages[].content`：消息内容，必填。
- `model`：模型名称，选填；不传则使用后端默认模型。
- `temperature`：生成随机性，选填；不传则使用后端默认值。

### 流式响应事件

```text
data: {"type":"delta","content":"你好"}
data: {"type":"delta","content":"，有什么可以帮你？"}
data: {"type":"done"}
```

### 错误事件

```text
data: {"type":"error","message":"模型调用失败，请稍后重试"}
```

### 验收标准

- 正常请求时能够持续返回 `delta` 事件。
- 完成时返回 `done` 事件。
- 异常时返回 `error` 事件。

## 10. 页面交互流程

## 10.1 首次打开

1. 页面加载。
2. 读取 localStorage。
3. 如果有历史会话，默认打开最近更新的会话。
4. 如果没有历史会话，创建一个空会话。

## 10.2 发送消息

1. 用户输入问题。
2. 用户按 Enter 或点击发送。
3. 前端创建用户消息。
4. 前端创建空助手消息。
5. 前端调用流式聊天接口。
6. 前端逐步追加助手回答。
7. 回答完成后保存会话。

## 10.3 停止生成

1. 用户点击停止按钮。
2. 前端中断当前请求。
3. 助手消息保留已经生成的内容。
4. 当前消息状态标记为完成或中断。
5. 输入框恢复可用。

## 14. 风险与应对

- 模型接口不可用：提供清晰错误提示，允许用户重新发送。
- 流式读取兼容问题：优先使用 Fetch ReadableStream，必要时切换 SSE。
- 上下文过长：只发送最近若干轮消息，后端增加基础裁剪。
- localStorage 容量有限：Demo 阶段限制会话数量或消息长度。
- API Key 泄漏风险：所有模型请求必须由后端发起，前端不能保存 API Key。
- 停止生成不完整：前端使用 AbortController 中断请求，后端监听连接关闭。
